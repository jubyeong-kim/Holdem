### [X] 1. 주어진 문제를 해결하는 완성된 코드가 제출되었나요?

- 문제에서 요구하는 기능이 정상적으로 작동하는지?
- 해당 조건을 만족하는 부분의 코드 및 결과물을 근거으로 첨부

**근거:**

- **보안 점검**: API 키, DB 접속 정보, 시크릿 키 등 민감 정보가 `.env` 파일로 분리되어 관리되고, `.gitignore`에 `.env` 및 `users.json`이 명시적으로 제외되어 있어 의도적인 보안 처리가 되어 있습니다. `auth.js`에서 비밀번호는 `scryptSync` + `salt` 방식의 해시 암호화로 저장되며, `timingSafeEqual`을 사용하여 타이밍 공격까지 방어하고 있습니다. IP당 계정 1개 제한과 로그인 시도 횟수 제한(5회/10분)을 통해 무차별 대입 공격도 차단되어 있습니다. `trust proxy` 설정으로 실제 클라이언트 IP를 정확히 읽어들여 IP 기반 보안 정책이 올바르게 동작합니다.

- **백엔드 동작**: `server.js`는 Express + Socket.IO 기반으로 실시간 멀티플레이 홀덤 서버를 구현하고 있습니다. REST API(`/api/register`, `/api/login`, `/api/logout`)로 인증을 처리하고, Socket.IO 이벤트(`host:create`, `player:join`, `player:action`, `control:act` 등)로 실시간 게임 상태를 동기화합니다. `poker.js`의 `Table` 클래스가 텍사스 홀덤의 모든 로직(덱 생성, 셔플, 블라인드 포스팅, 베팅 라운드 진행, 족보 판정, 팟 어워드)을 담당하며, `pokersolver` 라이브러리를 활용하여 정확한 족보 비교를 수행합니다. 특히 `view(forId)` 메서드가 각 플레이어에게 자신의 홀 카드만 노출하고 타 플레이어의 카드는 `??`로 숨기는 정보 은닉이 정확하게 구현되어 있습니다.

- **프론트엔드-백엔드 연동**: Socket.IO를 통해 서버의 `Table` 상태가 `state` 이벤트로 모든 클라이언트에게 브로드캐스팅되며, 각 프론트엔드 페이지(host.html, player.html, control.html)는 수신한 상태를 기반으로 실시간 UI를 렌더링합니다. QR 코드 자동 생성(`qrcode` 라이브러리)을 통해 모바일 폰으로 간편하게 접속할 수 있는 UX를 제공하며, LAN IP 자동 감지(`lanIP()`)와 Render 환경변수(`RENDER_EXTERNAL_URL`) 지원으로 로컬/클라우드 배포 모두에서 동작합니다.

### [X] 2. 핵심적이거나 복잡하고 이해하기 어려운 부분에 작성된 설명을 보고 해당 코드가 잘 이해되었나요?

- 해당 코드 블럭에 doc string/annotation/markdown이 달려 있는지 확인
- 해당 코드가 무슨 기능을 하는지, 왜 그렇게 짜여진건지, 작동 메커니즘이 뭔지 기술
- 주석을 보고 코드 이해가 잘 되었는지 확인
- 잘 작성되었다고 생각되는 부분을 근거으로 첨부합니다.

**근거:**

- `poker.js` 상단에 `"ponytail: main-pot only, no side pots"`라는 주석으로 현재 구현의 한계와 향후 확장 방향이 명확하게 명시되어 있습니다. `STREETS` 배열(`preflop`, `flop`, `turn`, `river`, `showdown`)을 통해 게임 진행 순서를 직관적으로 표현하였고, `advance(h)` 함수의 분기 로직(단일 플레이어 생존 시 즉시 어워드, 베팅 종료 시 다음 스트리트 진행)이 주석과 함께 이해하기 쉽게 작성되어 있습니다.

- `auth.js`의 무차별 대입 방지 로직(`MAX_FAILS = 5`, `WINDOW_MS = 10 * 60 * 1000`)은 상수와 함께 `fails` Map 구조의 동작 원리가 주석으로 설명되어 있습니다. `checkPw` 함수에서 `timingSafeEqual`을 사용하는 이유(타이밍 공격 방어)와 `hashPw` 함수의 salt 기반 해시 구조가 코드만으로도 명확하게 파악됩니다.

- `server.js`의 `broadcast(code)` 함수는 호스트 소켓, 각 플레이어 소켓, 컨트롤러 소켓 세 그룹에게 각각 `view(null)` 또는 `view(pid)`를 전달하는 구조로, `view(forId)`의 오버로딩이 어떻게 정보 은닉을 달성하는지 주석 없이도 의도가 명확합니다.

- `player.html`의 peek(squeeze) 기능은 `pointermove` 이벤트의 Y축 이동 거리로 `rotateY` 각도를 조절하는 로직(`(startY - e.clientY) * 1.5`)으로 구현되어 있으며, `perspective: 1000px`와 `backface-visibility: hidden` CSS를 활용한 3D 카드 뒤집기 효과가 직관적으로 이해됩니다.

- `server.js:12-19`의 `lanIP()` 함수는 `networkInterfaces()`를 순회하며 `192.168.*`, `10.*`, `172.*` 대역의 IPv4 주소를 우선 선택하는 로직이 정규표현식으로 명확하게 작성되어 있어 LAN 환경 감지 메커니즘을 쉽게 이해할 수 있습니다.

### [X] 3. 에러가 난 부분을 디버깅하여 "문제를 해결한 기록"을 남겼나요? 또는 "새로운 시도 및 추가 실험"을 해봤나요?

- 문제 원인 및 해결 과정을 잘 기록하였는지 확인
- 문제에서 요구하는 조건에 더해 추가적으로 수행한 나만의 시도, 실험이 기록되어 있는지 확인
- 잘 작성되었다고 생각되는 부분을 캡쳐해 근거으로 첨부합니다.

**근거:**

- **동시 접속/소켓 연결 해제 처리**: `server.js:161-172`의 `disconnect` 이벤트 핸들러에서 호스트, 컨트롤러, 플레이어 각 역할에 따라 분기하여 연결 해제를 처리합니다. 특히 플레이어는 `setConnected(playerId, false)`로 연결 상태를 표시하고 소켓 매핑을 정리하는 이중 처리가 되어 있어, 브라우저 새로고침이나 네트워크 끊김 시에도 게임 상태가 손상되지 않습니다.

- **게임 로직 예외 처리**: `poker.js`의 `act()` 메서드에서 `fold`, `check`, `call`, `raise` 각 액션에 대해 구체적인 에러 메시지를 반환합니다("네 차례가 아니야", "체크할 수 없어, 콜 금액이 있어", "칩이 부족해", "최소 N까지 레이즈해야 해"). `startHand()`에서 플레이어 2명 미만 시 시작 불가, `setBlinds()`에서 핸드 진행 중 변경 불가 등 방어적 코딩이 철저하게 되어 있습니다.

- **레이즈 로직의 minRaise 재설정**: `poker.js:151`에서 레이즈 시 `raiseBy >= h.minRaise` 조건으로 미니레이즈를 검증하고, 실제 레이즈 금액만큼 `h.minRaise`를 재설정하는 로직이 구현되어 있습니다. 이는 Texas Hold'em의 "raise to" 방식 베팅 규칙을 정확하게 반영한 것입니다.

- **프론트엔드 UI/UX 개선 시도**: `player.html`의 squeeze(카드 엿보기) 기능은 모바일 사용자를 위해 터치 드래그로 홀 카드를 확인할 수 있게 구현된 점이 돋보입니다. `control.html`의 `glow` 키프레임 애니메이션으로 현재 차례인 플레이어를 시각적으로 강조하는 점, 블라인드 프리셋 선택 다이얼로그(`BLIND_LEVELS` 배열)를 통해 간편하게 판돈을 조절할 수 있는 점 등 사용자 경험을 고려한 개선이 다수 포함되어 있습니다.

- **Render 배포 대비**: `render.yaml`과 `ORIGIN` 환경변수 처리(`PUBLIC_URL`, `RENDER_EXTERNAL_URL` 우선순위 체인)를 통해 로컬 개발과 클라우드 배포 환경을 모두 지원하도록 설계되어 있으며, `.env` 파일의 gitignore 처리로 비밀값 유출을 사전에 방지하는 아키텍처적 고려가 돋보입니다.

### [X] 5. 코드가 간결하고 효율적인가요?

- 파이썬 스타일 가이드 (PEP8)를 준수하였는지 확인
- 코드 중복을 최소화하고 범용적으로 사용할 수 있도록 모듈화(함수화) 했는지
- 잘 작성되었다고 생각되는 부분을 근거으로 첨부합니다.

**근거:**

- **모듈화**: 백엔드는 `poker.js`(게임 엔진), `auth.js`(인증), `server.js`(라우팅/소켓)로 역할에 따라 명확하게 분리되어 있으며, 프론트엔드는 `cards.js`를 공유 컴포넌트로 분리하여 host, player, control 세 페이지에서 중복 없이 재사용하고 있습니다.

- **코드 구조**: `Table` 클래스가 게임 상태 관리, 액션 처리, 뷰 생성을 하나의 캡슐화된 구조로 담당하며, `view(forId)` 메서드로 서버-클라이언트 간 데이터 전달을 단일 진입점으로 처리하는 설계가 깔끔합니다. `server.js`의 Socket.IO 이벤트 핸들러들이 각각 독립적인 콜백 구조를 가지되, 공통적인 `broadcast(code)` 호출 패턴으로 중복이 최소화되어 있습니다.

- **프론트엔드**: `player.html`, `control.html`, `host.html` 모두 동일한 CSS 변수 체계(`:root{--felt, --gold, --cream}`)를 공유하여 시각적 일관성을 유지하면서도, 각 페이지의 역할에 맞는 레이아웃과 인터랙션을 독립적으로 구현하고 있습니다. JavaScript 측에서도 `$ = (id) => document.getElementById(id)` 헬퍼와 Socket.IO `emit-callback` 패턴이 전체적으로 일관되게 사용되고 있어 가독성이 뛰어납니다.
